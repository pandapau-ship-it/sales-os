# ADR 007: Internationalisierung mit i18next von Anfang an

## Status
Accepted

## Kontext
Sales OS wird als SaaS für mehrere Märkte betrieben. UI-Texte nachträglich aus dem
JSX zu extrahieren ist teuer und fehleranfällig — Strings verstreuen sich über
Dutzende Komponenten, und jede neue Komponente zementiert die Hardcoding-Schuld
weiter. Gleichzeitig dürfen vom User eingegebene Inhalte (Kontakte, Notizen,
Nachrichten, Kurzakte) niemals übersetzt werden — nur die System-Oberfläche.

## Entscheidung
**i18n wird von Anfang an mit `i18next` + `react-i18next` eingebaut.** Alle UI-Strings
(Labels, Buttons, Menüs, Fehlermeldungen, Tooltips, System-Texte) liegen in
`src/locales/<lng>.json` und werden über `t("key")` aufgelöst — nie hardcodiert im JSX.

- Initialisierung ausschließlich in `src/lib/i18n.ts` (einziger Eintrittspunkt).
- Drei Sprachen: `de.json` · `en.json` · `es.json`. **Default + fallback: Deutsch.**
- EN/ES starten als DE-Kopie; fehlende Keys fallen automatisch auf DE zurück.
- Sprachwahl in **Settings → Allgemein**, persistiert in `localStorage` (`language`).
- Zugriff in Komponenten nur über `useLanguage()` / `useTranslation()`; Sprachwechsel
  nur über `setLanguage()` — kein direkter i18next-Import außerhalb von `lib/i18n.ts`.
- **User-Eingaben laufen nie durch `t()`** und werden nie übersetzt.

## Konsequenzen
**Positiv:**
- Keine Hardcoding-Schuld — neue Sprachen sind reine Übersetzungsarbeit, kein Code-Umbau.
- Ein zentraler Choke-Point (`lib/i18n.ts`) wie bei `aiCall()` / `notify()`.
- Klare Trennung System-UI ↔ User-Content verhindert versehentliche Übersetzung von Daten.

**Negativ:**
- Jede neue Komponente muss Keys in allen drei JSON-Dateien pflegen (Disziplin nötig).
- Bestehende Screens müssen schrittweise migriert werden (laufende Aufgabe, → CHECKLIST.md).
- Resources werden statisch gebündelt (kein Lazy-Loading pro Sprache) — bei vielen
  Sprachen später auf dynamisches Laden umstellen.

## Verworfene Alternativen
- **Eigene Lösung (Context + JSON-Map):** unnötig, i18next löst Pluralisierung,
  Interpolation, Fallback und Formatierung bereits robust.
- **react-intl (FormatJS):** schwergewichtiger, ICU-zentriert; i18next ist im
  React-Ökosystem verbreiteter und einfacher mit JSON-Dateien zu betreiben.
- **i18n erst später nachrüsten:** verworfen — genau das erzeugt die Hardcoding-Schuld,
  die wir vermeiden wollen.
